# Monorepo support

One `/knowledge/` at the repo root. Per-package subdirectories only under `specs/`. Decisions and ideas stay flat. Runbooks and architecture are **specs with a tag**, not separate folders.

## Why a single zone

Most monorepo knowledge is cross-cutting: a shared auth library is one spec; the deployment pipeline is one runbook-tagged spec; *"we use Postgres everywhere"* is one ADR. Per-package zones force duplication and require scanning multiple zones to answer one question. A single root zone with package-aware subdirs under `specs/` keeps the read path simple and lets cross-package relationships emerge through `related:`.

## Layout

```
/                                          ← repo root
├── knowledge/
│   ├── docs/
│   │   ├── specs/
│   │   │   ├── auth-service/
│   │   │   │   ├── jwt.md                       (tags: [auth, jwt])
│   │   │   │   ├── jwt-rotation.md              (tags: [auth, runbook])
│   │   │   │   └── topology.md                  (tags: [auth, architecture])
│   │   │   ├── inference-api/
│   │   │   │   ├── pipeline.md                  (tags: [inference])
│   │   │   │   └── gpu-oom.md                   (tags: [inference, runbook])
│   │   │   └── shared/                        ← cross-package specs
│   │   │       ├── api-conventions.md           (tags: [api])
│   │   │       └── overview.md                  (tags: [architecture])
│   │   └── decisions/                       ← ADRs flat, never per-package
│   │       ├── ADR-0001-monorepo-tooling.md
│   │       └── ADR-0003-ray-serve.md
│   ├── ideas/                               ← inbox, flat
│   └── INDEX.md                             ← auto-generated
├── packages/
│   ├── auth-service/
│   └── inference-api/
└── ...
```

Two things to internalize:

1. **Only `specs/` is package-partitioned.** Decisions/ideas stay flat — scope is encoded in content.
2. **`runbook` and `architecture` are reserved tags, not directories.** `/keep-ask` filters by tag; the file lives next to the related spec.

## Command behaviors

### `/keep-ask` — read path

Reads `INDEX.md`, filters by `description` / `domain` / `tags`. When the question names a package, that filter narrows results before tag/domain filters. Flat ADRs and `specs/shared/` surface alongside as cross-cutting context. If no package is named, searches across all subdirs.

### `/keep-compile` — diff classification

Infer affected packages from file paths:

- `packages/auth-service/src/...` → `auth-service`
- `packages/inference-api/src/...` → `inference-api`
- Both → cross-package (candidate for `specs/shared/`)

Behavioral and runbook-tagged specs go under the affected package's subdir. ADRs always flat. Architecture-tagged specs go under the package (package-local scope) or `specs/shared/` (cross-cutting).

For cross-package diffs:

- **Prefer `specs/shared/`** for the spec itself — don't duplicate per package.
- **List affected packages in `related:`** with path patterns:
  ```yaml
  related:
    - code:packages/auth-service/**
    - code:packages/inference-api/**
    - test:packages/{auth-service,inference-api}/**/*_test.go matching TestEnvelope*
  ```
- **ADRs affecting multiple packages stay flat.** `related:` lists multiple globs.

`/keep-compile` runs `build_index.py` after applying updates. The script walks `/knowledge/` and regenerates from frontmatter — no monorepo-specific logic needed.

### `/keep-check-drift`

Uses each file's `related:` patterns to determine if any spec/ADR is affected. Same matching works in monorepo and single-package — package-aware globs make scope-correct matching free.

### `/keep-govern`

Runs across the whole zone. Output groups by package subdir under `specs/`; flat artifacts (ADRs, ideas) listed separately.

## Decision tree — where does this content go?

| Content describes... | Location | Tags |
|---|---|---|
| Single package's behavior | `specs/<package>/<feature>.md` | `[<domain>]` |
| Behavior shared by multiple packages | `specs/shared/<topic>.md` | `[<domain>]` |
| Failure modes for one package | `specs/<package>/<failure-mode>.md` | `[<domain>, runbook]` |
| Failure modes crossing packages | `specs/shared/<failure-mode>.md` | `[<domain>, runbook]` |
| Internal topology of one package | `specs/<package>/topology.md` | `[<domain>, architecture]` |
| Cross-package topology | `specs/shared/overview.md` | `[architecture]` |
| Decision (any scope) | `decisions/ADR-NNNN-<slug>.md` | — (`type: adr`) |
| Half-formed proposal | `ideas/<YYYY-MM-DD>-<slug>.md` | `[<domain>]` |

When unclear, `/keep-compile` asks:

> *"This spec describes how the API responds to malformed JSON, which applies to both auth-service and inference-api. Put it in `specs/shared/` or in `specs/auth-service/` since that's where the change originated?"*

## `INDEX.md` in a monorepo

Auto-generated by `build_index.py`. The script groups behavioral specs by `specs/<package>/`; ADRs, ideas, and `specs/shared/` get their own sections. Example shape:

```md
## Packages (specs/)
- auth-service — 4 specs (2 behavioral, 1 runbook, 1 architecture)
- inference-api — 3 specs (2 behavioral, 1 runbook)
- shared — 2 specs

## All specs
| id | title | package | domain | tags | status |

## Decisions (flat)
| id | title | supersedes | status |

## Ideas (flat)
| id | title | created | status |
```

All derived from frontmatter. Never hand-edit. If the layout is wrong, change the script.

## Single-package fallback

In a non-monorepo:

- `specs/<package>/` → `specs/<area>/` (`specs/auth/`, `specs/payments/`).
- `specs/shared/` becomes unnecessary; cross-area concerns live where the dominant concern lives, with `related:` linking the rest.
- `## Packages` in `INDEX.md` collapses to `## Domains`. Script handles this.

All command behaviors apply unchanged.
